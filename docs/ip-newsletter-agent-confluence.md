## 한눈에 보기

> **IP Network 기술 동향(BGP, EVPN, DC 네트워크, 표준화 등)을 자동으로 수집·요약·검수해 뉴스레터 형태로 만들어주는 사내 Agent 도구입니다.**

- 📡 매일 흩어져 있는 RSS·벤더 블로그·IETF 표준 문서를 자동으로 모읍니다.
- 🤖 AI가 각 글을 카테고리 분류·한국어 요약·벤더 편향(홍보성) 검토까지 1차로 처리합니다.
- ✅ 담당자는 화면에서 **승인/반려만** 클릭하면 됩니다 — 최종 판단은 항상 사람이 합니다.
- 📰 승인된 기사만 모아 뉴스레터 초안을 자동 생성하고, 확인 후 발행합니다.

## 1. 왜 만들었나요? (배경)

IP Network 영역(백본, DCI, 데이터센터 네트워크, 보안, NetDevOps, 표준화 등)의 기술 동향은 여러 사이트에 흩어져 있어, 담당자가 매번 직접 찾아보고 정리해야 하는 부담이 있었습니다.

| 기존 문제 | 이 도구가 해결하는 방식 |
|-----------|--------------------------|
| 정보가 여러 사이트에 분산 | 등록된 소스를 자동 수집 + 필요 시 웹 검색으로 보강 |
| 벤더 블로그의 홍보성 표현 구분이 어려움 | AI가 1차로 편향 여부를 표시하고, 담당자가 최종 확인 |
| 분류·요약에 시간이 많이 듦 | AI가 카테고리 분류·한국어 요약·핵심 포인트를 자동 생성 |
| 표준화(IETF) 동향은 뉴스와 섞여 놓치기 쉬움 | IETF 동향만 별도로 모아 보여주는 전용 화면 제공 |

## 2. 무엇을 자동화하나요? (전체 흐름)

```
① 수집          여러 소스에서 기술 문서 자동 수집 (RSS/API/웹검색)
      ↓
② 1차 분석(AI)  카테고리 분류 · 한국어 요약 · 중요도 점수 · 편향 검토
      ↓
③ 자동 분류     점수가 높으면 "승인 대기", 애매하면 "추가 검토", 낮으면 자동 제외
      ↓
④ 사람 검수     담당자가 화면에서 최종 승인 / 반려 (필수 단계)
      ↓
⑤ 뉴스레터 생성  승인된 기사만 모아 초안 자동 작성
      ↓
⑥ 발행          담당자가 최종 확인 후 발행 확정
```

> 핵심 원칙: **AI는 초안까지만 만들고, 뉴스레터에 실제로 포함할지 최종 결정은 항상 사람이 합니다.**

## 3. 어떻게 만들어졌나요? (Multi-Agent 구조 쉽게 이해하기)

이 도구는 하나의 거대한 프로그램이 혼자 처리하는 방식이 아니라, **각자 역할이 다른 여러 "AI 담당자"가 순서대로 협업**하는 구조로 만들어졌습니다. 이런 구조를 **Multi-Agent(멀티 에이전트) 시스템**이라고 부릅니다.

> 💡 **비유로 이해하기:** 신문사 편집국을 떠올려보세요. 취재기자가 소재를 모으고, 데스크가 1차로 검토·분류하고, 전문 리포터가 특정 분야를 따로 취재하고, 편집장이 최종 승인하고, 조판 담당이 지면을 완성합니다. 이 도구도 똑같은 방식으로, AI 담당자들이 각자 맡은 역할만 수행하고 다음 담당자에게 결과물을 넘깁니다.

### 담당 Agent 소개

| 담당 (Agent) | 사람으로 비유하면 | 실제로 하는 일 |
|---|---|---|
| 🔎 **Research Agent** | 취재기자 | 등록된 소스에서 기사를 수집하고, 부족한 분야는 웹 검색으로 추가 취재 |
| 📊 **Analysis Agent** | 데스크 (1차 검토자) | 기사를 읽고 카테고리 분류·한국어 요약·중요도·벤더 편향 여부를 판단 |
| 📡 **Standards Radar Agent** | 전문 분야 리포터 | IETF 표준화 문서만 따로 모니터링해서 별도로 정리 |
| 🧑‍💼 **Human Agent** (담당자 본인) | 편집장 | AI의 판단 결과를 최종 확인하고 승인/반려를 결정 (사람이 하는 유일한 필수 단계) |
| ✍️ **Editor Agent** | 조판·교열 담당 | 승인된 기사들을 모아 뉴스레터 형태로 최종 조립 |

### 담당자들의 작업 순서는 누가 정하나요?

이 담당자들이 **어떤 순서로, 어떤 조건에서** 일할지 관리하는 "팀장" 역할을 **Orchestrator(오케스트레이터)** 가 맡습니다. Orchestrator는 `LangGraph`라는 기술로 만들어졌으며, 아래처럼 흐름을 조율합니다.

```
[Research Agent]        여러 소스에서 기사 수집 (+ 필요 시 웹 검색)
       ↓
[Analysis Agent]        카테고리 분류 · 한국어 요약 · 편향 검토
       ↓
[Standards Radar Agent] IETF 표준화 동향만 별도로 정리 (병행 진행)
       ↓
  공유 저장소(Vault)에 결과 저장   ← 담당자들이 서로 결과물을 넘겨받는 "공유 폴더"
       ↓
[Human Agent] 사람이 화면에서 최종 승인/반려 결정  ★ 필수 단계
       ↓
[Editor Agent]           승인된 기사만 모아 뉴스레터 초안 완성
```

각 담당자는 자신의 결과물을 **Vault**라는 공유 저장소(문서 폴더)에 기록하고, 다음 담당자는 그 저장소에서 필요한 내용을 읽어 이어서 작업합니다. 사람이 하는 검수(Human Agent)도 이 흐름의 정식 단계 중 하나로 포함되어 있어서, AI가 아무리 많은 판단을 하더라도 **뉴스레터에 실제로 실릴지 여부는 항상 사람이 최종 결정**합니다.

### 왜 AI 하나가 다 처리하지 않고 여러 Agent로 나눴나요?

- **역할이 명확해져 결과를 신뢰하기 쉬워집니다.** 어떤 담당자가 무슨 판단을 했는지 단계별로 확인할 수 있습니다.
- **문제가 생겨도 원인을 빠르게 찾을 수 있습니다.** 예를 들어 편향 판단이 이상하면 Analysis Agent 단계만 확인하면 됩니다.
- **사람의 개입(HITL)을 원하는 지점에 정확히 배치할 수 있습니다.** 전 과정을 AI에 맡기지 않고, 뉴스레터 반영 여부처럼 중요한 결정 앞에 사람의 확인 단계를 넣을 수 있습니다.

## 4. 주요 기능

| 기능 | 설명 |
|------|------|
| **자동 수집** | APNIC, RIPE, OCP, Cisco, Nokia 등 사전 등록된 소스를 정기적으로 수집 |
| **웹 검색 보강** | 등록 소스만으로 부족한 카테고리는 AI가 웹 검색으로 최신 동향을 추가 탐색 |
| **AI 1차 분석** | 카테고리 분류(9종), 한국어 요약, 핵심 포인트, 중요도(1~5점), 벤더 편향 여부 자동 산출 |
| **사람 검수 (HITL)** | 담당자가 목록에서 기사를 확인하고 승인/반려 — 최종 뉴스레터 반영 여부 결정 |
| **IETF 표준 레이더** | BGP, EVPN, 세그먼트 라우팅 등 관련 IETF 워킹그룹의 표준화 동향을 별도로 요약 |
| **뉴스레터 자동 생성** | 승인된 기사를 카테고리별로 정리한 뉴스레터 초안 자동 작성 |
| **발행 관리** | 발행 이력 관리, 중복(재발행) 자동 차단, 호별 아카이브 보관 |

## 5. 사용 방법 (화면 구성)

도구는 웹 화면(Streamlit) 하나로 모든 작업을 처리합니다. 실행하면 아래 5개 탭이 순서대로 보입니다.

| 순서 | 화면 | 하는 일 |
|------|------|---------|
| 1 | 📥 **데이터 수집** | 수집할 소스를 선택하고 실행 버튼을 누르면 자동 수집·1차 분석까지 진행됩니다 |
| 2 | 📋 **기사 검토** | AI가 분석한 기사 목록을 확인하고 **승인/반려**를 결정합니다 (핵심 업무 화면) |
| 3 | 📰 **뉴스레터** | 승인된 기사로 뉴스레터 초안을 생성하고, 검토 후 **발행 확정**합니다 |
| 4 | 📦 **아카이브** | 지난 호 발행 뉴스레터와 사용된 기사를 다시 찾아볼 수 있습니다 |
| 5 | ⚙️ **운영콘솔** | (선택) 수집 상세 로그, 표준화(IETF) 동향 전체, 시스템 실행 기록 확인 |

### 실제 사용 순서 (일반적인 하루 업무 흐름)

1. **📥 데이터 수집** 탭에서 [수집 실행] 클릭 → 몇 분 내 자동으로 기사 수집 및 1차 분석 완료
2. **📋 기사 검토** 탭에서 "승인 대기" 큐를 확인 → 기사별 요약·편향 표시를 보고 **승인** 또는 **반려**
3. **📰 뉴스레터** 탭에서 [뉴스레터 생성] 클릭 → 초안 확인 → 이상 없으면 **발행 확정**
4. 필요 시 **📦 아카이브** 탭에서 지난 호를 다시 확인

> 💡 승인/반려는 되돌릴 수 있으며, 한번 발행된 기사는 자동으로 재사용이 차단되어 중복 걱정 없이 운영할 수 있습니다.

## 6. 기사 검토 시 확인 포인트

기사 검토 화면에서 각 기사는 아래 정보와 함께 표시됩니다.

| 표시 항목 | 의미 |
|-----------|------|
| ★ 중요도 (1~5) | AI가 판단한 뉴스레터 포함 중요도 |
| 카테고리 | Routing, Backbone, DC Network, 보안, NetDevOps, 표준화 등 9개 분류 |
| 편향 표시 (🟢🟡🔴) | 벤더 홍보성 표현 위험도 — 🔴/🟡는 특히 유의해서 검토 |
| 요약 / 핵심 포인트 | AI가 생성한 한국어 3~5줄 요약 |
| 검토 상태 | 승인 대기 / 추가 검토 필요 / 자동 제외 |

**승인 대기**로 분류된 기사는 AI 판단 신뢰도가 높은 기사이고, **추가 검토**는 애매하거나 검증이 더 필요한 기사입니다. 최종 승인 여부는 항상 담당자가 결정합니다.

## 7. 자주 묻는 질문 (FAQ)

**Q1. AI가 잘못된 내용을 승인하면 어떻게 되나요?**  
A. AI는 초안만 만들고, 화면에서 **사람이 최종 승인**해야만 뉴스레터에 포함됩니다. 승인 전에 요약·편향 표시를 확인하는 절차가 반드시 있습니다.

**Q2. 같은 기사가 다음 호에 또 나오지는 않나요?**  
A. 한 번 발행된 기사는 자동으로 "발행 완료" 목록에 등록되어, 이후 수집·검토·뉴스레터 생성 단계에서 자동으로 제외됩니다.

**Q3. 벤더(제조사) 블로그 내용은 믿을 수 있나요?**  
A. 벤더 소스는 별도로 표시되며, AI가 홍보성 표현 위험도(편향 점수)를 함께 보여줍니다. 검토 시 참고해 판단할 수 있습니다.

**Q4. IETF 표준화 동향은 뉴스레터에 어떻게 반영되나요?**  
A. IETF 표준 동향은 일반 뉴스와 별도로 관리되며, 승인된 기사와 관련 있는 경우에만 뉴스레터에 참고 정보로 자동 연결됩니다.

**Q5. 실행에 얼마나 걸리나요?**  
A. 수집~1차 분석까지 보통 수 분 내 완료됩니다. 소스 수·웹 검색 포함 여부에 따라 달라질 수 있습니다.

**Q6. 담당자가 바뀌면 어떻게 인수인계하나요?**  
A. 본 페이지의 "사용 방법"과 "기사 검토 시 확인 포인트"만 숙지하면 바로 운영 가능합니다. 상세 설정 변경은 개발 담당자에게 문의하세요.

## 9. 용어 설명

| 용어 | 설명 |
|------|------|
| Multi-Agent (멀티 에이전트) | 하나의 AI가 아니라, 역할이 다른 여러 AI 담당자가 나눠서 협업하는 구조 |
| Agent (에이전트) | 특정 역할(수집·분석·검수·조립 등)을 전담하는 AI 담당자 단위 |
| Orchestrator (오케스트레이터) | 여러 Agent의 작업 순서와 흐름을 관리하는 "팀장" 역할의 관리 로직 (LangGraph로 구현) |
| HITL (Human-in-the-Loop) | AI 처리 결과를 사람이 검수·승인하는 절차 |
| Vault | 수집~발행까지의 모든 산출물이 저장되는 공유 폴더(Obsidian 형식) — Agent 간 결과물 전달 통로 |
| 승인 대기 / 추가 검토 / 자동 제외 | AI 신뢰도 점수에 따른 자동 분류 큐 |
| 표준화 레이더 (IETF Radar) | IETF 표준 워킹그룹의 최신 동향을 모아 보여주는 참고 화면 |
| 발행 registry | 이미 발행된 기사 이력을 저장해 중복 발행을 막는 저장소 |
